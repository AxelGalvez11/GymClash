## Design Principles

- **Client-owned fields** can be written by the authenticated user (via RLS policies). These represent user intent: display names, workout logs, report descriptions.
- **Server-derived fields** are computed by Supabase Edge Functions, database triggers, or backend jobs. Clients cannot write to these directly. Examples: scores, ranks, validation results.
- **Mutable fields** can change after creation (e.g., `display_name`).
- **Authoritative fields** are the system's source of truth and must never be set by the client (e.g., `xp`, `rank`, `final_score`).
- All timestamps use `timestamptz` (UTC).
- All primary keys are `uuid` with `gen_random_uuid()` defaults.
- JSONB columns are used for semi-structured data that varies by workout type or evidence type, avoiding excessive table proliferation.

## Entity Definitions

### 1. `users`

> Managed entirely by **Supabase Auth**. Do not create this table manually.

| Field | Type | Notes |
|-------|------|-------|
| `id` | `uuid` PK | Generated by Supabase Auth |
| `email` | `text` | Unique, managed by Auth |
| `created_at` | `timestamptz` | Auto-set by Auth |
| `updated_at` | `timestamptz` | Auto-set by Auth |

**Ownership:** All fields are server-derived (Supabase Auth).

**Notes:**
- Referenced as `auth.users` in foreign keys.
- The `profiles` table extends this with app-specific data.
- Never store app-level data directly in `auth.users.raw_user_meta_data`; use `profiles` instead.

---

### 2. `profiles`

> Player identity, progression, and rank. Created automatically via a database trigger on `auth.users` insert.

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK, FK -> `auth.users(id)` ON DELETE CASCADE | -- | Server-derived (matches auth user id) |
| `display_name` | `text` NOT NULL | `''` | **Client-owned**, mutable |
| `avatar_url` | `text` | `NULL` | **Client-owned**, mutable |
| `rank` | `enum('bronze','silver','gold','platinum','diamond','champion')` | `'bronze'` | Server-derived, authoritative |
| `xp` | `integer` NOT NULL | `0` | Server-derived, authoritative |
| `level` | `integer` NOT NULL | `1` | Server-derived, authoritative |
| `current_streak` | `integer` NOT NULL | `0` | Server-derived, authoritative |
| `longest_streak` | `integer` NOT NULL | `0` | Server-derived, authoritative |
| `last_workout_date` | `date` | `NULL` | Server-derived, authoritative |
| `subscription_tier` | `enum('free','premium')` | `'free'` | Server-derived (set via purchase verification) |
| `created_at` | `timestamptz` NOT NULL | `now()` | Server-derived |
| `updated_at` | `timestamptz` NOT NULL | `now()` | Server-derived (trigger) |

**State rules:**
- `display_name` and `avatar_url` are the only client-writable columns. RLS policy restricts updates to these two fields only.
- `rank` is recomputed from `xp` thresholds by a server function.
- `level` is derived from `xp` (e.g., `floor(xp / 1000) + 1`).
- `current_streak` and `longest_streak` are updated by the workout validation pipeline. `last_workout_date` gates streak logic.
- `subscription_tier` is updated only by a server-side webhook handler (App Store / Google Play receipt verification).

**Trigger:** `updated_at` is refreshed on every row update via a `BEFORE UPDATE` trigger.

### 3. `workouts`

> The core activity record. Submitted by the client, scored and validated by the server.

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK | `gen_random_uuid()` | Server-derived |
| `user_id` | `uuid` NOT NULL, FK -> `auth.users(id)` ON DELETE CASCADE | -- | Server-derived (from auth context) |
| `type` | `enum('strength','scout')` NOT NULL | -- | **Client-owned**, immutable after submit |
| `status` | `enum('draft','submitted','validated','rejected')` NOT NULL | `'draft'` | Server-derived, authoritative |
| `started_at` | `timestamptz` NOT NULL | -- | **Client-owned**, immutable after submit |
| `completed_at` | `timestamptz` | `NULL` | **Client-owned**, immutable after submit |
| `duration_seconds` | `integer` | `NULL` | **Client-owned**, immutable after submit |
| `sets` | `jsonb` | `NULL` | **Client-owned**, immutable after submit |
| `route_data` | `jsonb` | `NULL` | **Client-owned**, immutable after submit |
| `raw_score` | `numeric` | `NULL` | Server-derived, authoritative |
| `final_score` | `numeric` | `NULL` | Server-derived, authoritative |
| `confidence_score` | `numeric` CHECK (0.0 <= val <= 1.0) | `NULL` | Server-derived, authoritative |
| `validation_status` | `enum('accepted','accepted_with_low_confidence','held_for_review','excluded_from_clan_score','rejected')` | `NULL` | Server-derived, authoritative |
| `idempotency_key` | `text` UNIQUE NOT NULL | -- | **Client-owned**, immutable |
| `source` | `enum('manual','sensor','wearable')` NOT NULL | `'manual'` | **Client-owned**, immutable after submit |
| `created_at` | `timestamptz` NOT NULL | `now()` | Server-derived |
| `updated_at` | `timestamptz` NOT NULL | `now()` | Server-derived (trigger) |

**JSONB schemas:**

`sets` (for `type = 'strength'`):
```json
[
  {
    "exercise": "bench_press",
    "sets": 3,
    "reps": 10,
    "weight_kg": 80.0
  }
]
```

`route_data` (for `type = 'scout'`):
```json
{
  "distance_km": 5.2,
  "avg_pace_min_per_km": 5.5,
  "elevation_gain_m": 120,
  "route_points": [
    { "lat": 19.4326, "lng": -99.1332, "ts": "2026-03-30T08:00:00Z" }
  ]
}
```

**State rules:**
- Client inserts rows with `status = 'draft'` or `'submitted'`. Once `status` leaves `'draft'`, client-owned fields become immutable (enforced by RLS + trigger).
- `idempotency_key` prevents duplicate submissions from flaky networks. The client generates a UUID before submission.
- `raw_score` is computed from the workout data. `final_score` applies multipliers (season config, diminishing returns). `confidence_score` aggregates evidence trust levels.
- `validation_status` is set by the validation pipeline after all `workout_validations` checks complete.

**Constraint:** `CHECK (type = 'strength' AND sets IS NOT NULL) OR (type = 'scout' AND route_data IS NOT NULL)` -- ensures the correct JSONB column is populated for the workout type.

### 4. `workout_evidence`

> Supporting evidence attached to a workout to increase trust / confidence scoring.

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK | `gen_random_uuid()` | Server-derived |
| `workout_id` | `uuid` NOT NULL, FK -> `workouts(id)` ON DELETE CASCADE | -- | **Client-owned** (at creation) |
| `evidence_type` | `enum('gps_route','heart_rate_log','accelerometer_summary','photo','wearable_sync','manual_note')` NOT NULL | -- | **Client-owned**, immutable |
| `data` | `jsonb` NOT NULL | -- | **Client-owned**, immutable |
| `trust_level` | `integer` CHECK (1 <= val <= 5) | `NULL` | Server-derived, authoritative |
| `created_at` | `timestamptz` NOT NULL | `now()` | Server-derived |

**State rules:**
- Client submits evidence; it is immutable after creation.
- `trust_level` is computed server-side based on `evidence_type` and data analysis (e.g., GPS consistency, heart rate plausibility).
- `trust_level` feeds into the parent workout's `confidence_score`.

---

### 5. `workout_validations`

> Individual validation check results for a workout. Fully server-derived.

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK | `gen_random_uuid()` | Server-derived |
| `workout_id` | `uuid` NOT NULL, FK -> `workouts(id)` ON DELETE CASCADE | -- | Server-derived |
| `validation_type` | `text` NOT NULL | -- | Server-derived |
| `passed` | `boolean` NOT NULL | -- | Server-derived |
| `confidence_impact` | `numeric` NOT NULL | `0` | Server-derived |
| `reason_code` | `text` | `NULL` | Server-derived |
| `details` | `jsonb` | `NULL` | Server-derived |
| `created_at` | `timestamptz` NOT NULL | `now()` | Server-derived |

**State rules:**
- All columns are server-derived. Clients have read-only access to their own workout validations.
- One workout can have many validation records (e.g., `duration_check`, `gps_consistency`, `heart_rate_plausibility`, `repetition_rate_check`).
- `confidence_impact` is a numeric modifier (positive or negative) applied to the workout's `confidence_score`.
- `reason_code` provides a machine-readable failure reason (e.g., `DURATION_TOO_SHORT`, `GPS_TELEPORT_DETECTED`).

### 6. `clans`

> Competitive groups. A clan participates in seasonal wars.

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK | `gen_random_uuid()` | Server-derived |
| `name` | `text` UNIQUE NOT NULL | -- | **Client-owned** (leader), mutable |
| `tag` | `text` UNIQUE NOT NULL, CHECK (length between 3-6, alphanumeric) | -- | **Client-owned** (leader), immutable after creation |
| `description` | `text` | `''` | **Client-owned** (leader/officer), mutable |
| `leader_id` | `uuid` NOT NULL, FK -> `auth.users(id)` | -- | Server-derived (set on creation, transferable) |
| `max_members` | `integer` NOT NULL | `30` | Server-derived |
| `member_count` | `integer` NOT NULL | `1` | Server-derived, authoritative |
| `created_at` | `timestamptz` NOT NULL | `now()` | Server-derived |
| `updated_at` | `timestamptz` NOT NULL | `now()` | Server-derived (trigger) |

**State rules:**
- `member_count` is maintained by a trigger on `clan_memberships` (increment on insert, decrement on delete). Never set by the client.
- `tag` is immutable after creation (used in display badges, war history).
- `name` can be changed by the leader (rate-limited via Edge Function).
- `max_members` can be increased for premium clans via server logic.

---

### 7. `clan_memberships`

> Junction table linking users to clans with roles.

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK | `gen_random_uuid()` | Server-derived |
| `clan_id` | `uuid` NOT NULL, FK -> `clans(id)` ON DELETE CASCADE | -- | Server-derived |
| `user_id` | `uuid` NOT NULL, FK -> `auth.users(id)` ON DELETE CASCADE | -- | Server-derived |
| `role` | `enum('leader','officer','member')` NOT NULL | `'member'` | Server-derived |
| `joined_at` | `timestamptz` NOT NULL | `now()` | Server-derived |

**Constraints:**
- `UNIQUE(clan_id, user_id)` -- a user cannot join the same clan twice.
- `UNIQUE(user_id)` -- a user can only be in **one clan at a time**. This is the critical constraint.

**State rules:**
- Joining and leaving clans goes through Edge Functions that enforce business rules (cooldown periods, war lockouts).
- `role` is managed server-side. Promotion/demotion is a leader action routed through an Edge Function.

---

### 8. `clan_wars`

> A weekly matchup between two clans within a season.

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK | `gen_random_uuid()` | Server-derived |
| `season_id` | `uuid` NOT NULL, FK -> `seasons(id)` | -- | Server-derived |
| `clan_a_id` | `uuid` NOT NULL, FK -> `clans(id)` | -- | Server-derived |
| `clan_b_id` | `uuid` NOT NULL, FK -> `clans(id)` | -- | Server-derived |
| `week_number` | `integer` NOT NULL | -- | Server-derived |
| `status` | `enum('scheduled','active','completed')` NOT NULL | `'scheduled'` | Server-derived |
| `started_at` | `timestamptz` | `NULL` | Server-derived |
| `ended_at` | `timestamptz` | `NULL` | Server-derived |
| `winner_clan_id` | `uuid`, FK -> `clans(id)` | `NULL` | Server-derived |
| `clan_a_score` | `jsonb` | `NULL` | Server-derived |
| `clan_b_score` | `jsonb` | `NULL` | Server-derived |
| `created_at` | `timestamptz` NOT NULL | `now()` | Server-derived |

**JSONB score breakdown:**
```json
{
  "total": 4520,
  "strength_points": 2800,
  "scout_points": 1720,
  "participation_bonus": 200,
  "contributor_count": 18
}
```

**State rules:**
- All fields are server-derived. Wars are created by a scheduled job (cron) that runs matchmaking at the start of each season week.
- `status` transitions: `scheduled` -> `active` (when `started_at` arrives) -> `completed` (when `ended_at` arrives or manually finalized).
- `winner_clan_id` is set when status becomes `completed`. `NULL` indicates a draw.

**Constraint:** `CHECK (clan_a_id <> clan_b_id)` -- a clan cannot fight itself.

---

### 9. `clan_war_contributions`

> Links a validated workout to a specific war's score. Tracks diminishing returns.

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK | `gen_random_uuid()` | Server-derived |
| `war_id` | `uuid` NOT NULL, FK -> `clan_wars(id)` ON DELETE CASCADE | -- | Server-derived |
| `user_id` | `uuid` NOT NULL, FK -> `auth.users(id)` | -- | Server-derived |
| `clan_id` | `uuid` NOT NULL, FK -> `clans(id)` | -- | Server-derived |
| `workout_id` | `uuid` NOT NULL, FK -> `workouts(id)` | -- | Server-derived |
| `contribution_points` | `numeric` NOT NULL | -- | Server-derived, authoritative |
| `created_at` | `timestamptz` NOT NULL | `now()` | Server-derived |

**State rules:**
- All fields are server-derived. Created by the workout validation pipeline when a workout passes validation during an active war.
- `contribution_points` applies diminishing returns: the Nth workout from the same user in the same war contributes less than the (N-1)th. Formula is server-side only.
- `UNIQUE(workout_id)` -- a workout can only contribute to one war.

### 10. `reports`

> User-submitted reports for suspected cheating or inappropriate behavior.

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK | `gen_random_uuid()` | Server-derived |
| `reporter_id` | `uuid` NOT NULL, FK -> `auth.users(id)` | -- | Server-derived (from auth context) |
| `reported_user_id` | `uuid` NOT NULL, FK -> `auth.users(id)` | -- | **Client-owned**, immutable |
| `reported_workout_id` | `uuid`, FK -> `workouts(id)` ON DELETE SET NULL | `NULL` | **Client-owned**, immutable |
| `category` | `enum('impossible_stats','suspected_spoofing','inappropriate_behavior','other')` NOT NULL | -- | **Client-owned**, immutable |
| `description` | `text` NOT NULL | -- | **Client-owned**, immutable |
| `status` | `enum('pending','investigating','resolved','dismissed')` NOT NULL | `'pending'` | Server-derived, authoritative |
| `resolution_notes` | `text` | `NULL` | Server-derived |
| `created_at` | `timestamptz` NOT NULL | `now()` | Server-derived |
| `resolved_at` | `timestamptz` | `NULL` | Server-derived |

**State rules:**
- Client can insert reports but cannot update or delete them.
- `status` and `resolution_notes` are managed by admin tooling / Edge Functions.
- Rate limit: max 5 reports per user per day (enforced in Edge Function).

### 11. `appeals`

> User appeals against rejected workouts or penalties.

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK | `gen_random_uuid()` | Server-derived |
| `user_id` | `uuid` NOT NULL, FK -> `auth.users(id)` | -- | Server-derived (from auth context) |
| `workout_id` | `uuid` NOT NULL, FK -> `workouts(id)` | -- | **Client-owned**, immutable |
| `reason` | `text` NOT NULL | -- | **Client-owned**, immutable |
| `evidence_urls` | `text[]` | `'{}'` | **Client-owned**, immutable |
| `status` | `enum('pending','under_review','approved','denied')` NOT NULL | `'pending'` | Server-derived, authoritative |
| `reviewer_notes` | `text` | `NULL` | Server-derived |
| `created_at` | `timestamptz` NOT NULL | `now()` | Server-derived |
| `resolved_at` | `timestamptz` | `NULL` | Server-derived |

**State rules:**
- Client can create one appeal per workout (enforced via `UNIQUE(workout_id, user_id)`).
- `evidence_urls` point to files uploaded to Supabase Storage. The client uploads first, then submits the appeal with the resulting URLs.
- If approved, the workout's `validation_status` is updated and scores are recalculated.

### 12. `cosmetic_inventory`

> Cosmetic items earned or purchased by users (badges, themes, titles, clan banners).

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK | `gen_random_uuid()` | Server-derived |
| `user_id` | `uuid` NOT NULL, FK -> `auth.users(id)` ON DELETE CASCADE | -- | Server-derived |
| `cosmetic_type` | `enum('badge','theme','title','clan_banner')` NOT NULL | -- | Server-derived |
| `cosmetic_id` | `text` NOT NULL | -- | Server-derived |
| `acquired_via` | `enum('season_reward','purchase','achievement')` NOT NULL | -- | Server-derived |
| `acquired_at` | `timestamptz` NOT NULL | `now()` | Server-derived |

**Constraints:**
- `UNIQUE(user_id, cosmetic_type, cosmetic_id)` -- a user cannot own the same cosmetic twice.

**State rules:**
- All fields are server-derived. Cosmetics are granted by:

  - Achievement triggers (database trigger or Edge Function when conditions are met).
- Clients have read-only access to their own inventory.

### 13. `seasons`

> Time-bounded competitive periods with configurable scoring rules.

| Field | Type | Default | Ownership |
|-------|------|---------|-----------|
| `id` | `uuid` PK | `gen_random_uuid()` | Server-derived |
| `name` | `text` NOT NULL | -- | Server-derived |
| `number` | `integer` UNIQUE NOT NULL | -- | Server-derived |
| `started_at` | `timestamptz` NOT NULL | -- | Server-derived |
| `ended_at` | `timestamptz` NOT NULL | -- | Server-derived |
| `status` | `enum('upcoming','active','completed')` NOT NULL | `'upcoming'` | Server-derived |
| `config` | `jsonb` NOT NULL | `'{}'` | Server-derived |
| `created_at` | `timestamptz` NOT NULL | `now()` | Server-derived |

**JSONB config example:**
```json
{
  "scoring_multipliers": {
    "strength": 1.0,
    "scout": 1.2
  },
  "diminishing_returns_curve": "logarithmic",
  "max_daily_contributions_per_user": 3,
  "special_rules": [
    { "type": "double_xp_weekend", "dates": ["2026-04-11", "2026-04-12"] }
  ]
}
```

**State rules:**
- All fields are server-derived. Seasons are created by admins.
- `status` transitions: `upcoming` -> `active` (cron at `started_at`) -> `completed` (cron at `ended_at`).
- Only one season can be `active` at a time (enforced by partial unique index or application logic).
- `config` drives scoring formulas in the workout validation pipeline and war scoring.

## Index Recommendations

### Primary lookup patterns

```sql
-- profiles: looked up by user id (PK covers this)

-- workouts: user's workout history, ordered by date
CREATE INDEX idx_workouts_user_id_created ON workouts (user_id, created_at DESC);

-- workouts: idempotency check on submission
-- (UNIQUE constraint on idempotency_key already creates this index)

-- workouts: server pipeline queries by status
CREATE INDEX idx_workouts_status ON workouts (status) WHERE status IN ('submitted', 'validated');

-- workout_evidence: fetch evidence for a workout
CREATE INDEX idx_workout_evidence_workout_id ON workout_evidence (workout_id);

-- workout_validations: fetch validations for a workout
CREATE INDEX idx_workout_validations_workout_id ON workout_validations (workout_id);

-- clan_memberships: list members of a clan
CREATE INDEX idx_clan_memberships_clan_id ON clan_memberships (clan_id);

-- clan_memberships: find which clan a user belongs to
-- (UNIQUE constraint on user_id already creates this index)

-- clan_wars: list wars in a season
CREATE INDEX idx_clan_wars_season_id ON clan_wars (season_id, week_number);

-- clan_wars: find wars for a specific clan
CREATE INDEX idx_clan_wars_clan_a ON clan_wars (clan_a_id, status);
CREATE INDEX idx_clan_wars_clan_b ON clan_wars (clan_b_id, status);

-- clan_war_contributions: aggregate scores for a war
CREATE INDEX idx_war_contributions_war_id ON clan_war_contributions (war_id, clan_id);

-- clan_war_contributions: user's contributions
CREATE INDEX idx_war_contributions_user_id ON clan_war_contributions (user_id);

-- reports: admin queue
CREATE INDEX idx_reports_status ON reports (status) WHERE status = 'pending';

-- appeals: admin queue
CREATE INDEX idx_appeals_status ON appeals (status) WHERE status IN ('pending', 'under_review');

-- cosmetic_inventory: user's collection
CREATE INDEX idx_cosmetic_inventory_user_id ON cosmetic_inventory (user_id);

-- seasons: find the active season
CREATE UNIQUE INDEX idx_seasons_active ON seasons (status) WHERE status = 'active';
```

### Partial indexes

The `WHERE` clauses above create partial indexes, which are smaller and faster for the specific query patterns they serve (e.g., only pending reports, only active seasons).

## Row Level Security (RLS)

All tables must have RLS enabled. Policies follow the principle of least privilege.

### `profiles`

```sql
-- Anyone can read any profile (public leaderboards, clan rosters)
CREATE POLICY "profiles_select" ON profiles FOR SELECT USING (true);

-- Users can update only their own profile, only display_name and avatar_url
CREATE POLICY "profiles_update" ON profiles FOR UPDATE
  USING (auth.uid() = id)
  WITH CHECK (auth.uid() = id);
-- Note: Use a column-level grant or trigger to restrict updates to only
-- display_name and avatar_url. RLS alone cannot restrict which columns change.
```

### `workouts`

```sql
-- Users can read their own workouts
CREATE POLICY "workouts_select_own" ON workouts FOR SELECT
  USING (auth.uid() = user_id);

-- Clan members can read validated workouts of clanmates (for war contribution visibility)
CREATE POLICY "workouts_select_clan" ON workouts FOR SELECT
  USING (
    validation_status = 'accepted'
    AND user_id IN (
      SELECT cm2.user_id FROM clan_memberships cm1
      JOIN clan_memberships cm2 ON cm1.clan_id = cm2.clan_id
      WHERE cm1.user_id = auth.uid()
    )
  );

-- Users can insert their own workouts
CREATE POLICY "workouts_insert" ON workouts FOR INSERT
  WITH CHECK (auth.uid() = user_id);

-- Users can update their own draft workouts (submit them)
CREATE POLICY "workouts_update_draft" ON workouts FOR UPDATE
  USING (auth.uid() = user_id AND status = 'draft')
  WITH CHECK (auth.uid() = user_id);
```

### `workout_evidence`

```sql
-- Users can read evidence for their own workouts
CREATE POLICY "evidence_select" ON workout_evidence FOR SELECT
  USING (workout_id IN (SELECT id FROM workouts WHERE user_id = auth.uid()));

-- Users can insert evidence for their own workouts
CREATE POLICY "evidence_insert" ON workout_evidence FOR INSERT
  WITH CHECK (workout_id IN (SELECT id FROM workouts WHERE user_id = auth.uid()));
```

### `workout_validations`

```sql
-- Users can read validations for their own workouts (transparency)
CREATE POLICY "validations_select" ON workout_validations FOR SELECT
  USING (workout_id IN (SELECT id FROM workouts WHERE user_id = auth.uid()));

-- No insert/update/delete for clients (server-only via service_role key)
```

### `clans`

```sql
-- Anyone can read clans (search, browse)
CREATE POLICY "clans_select" ON clans FOR SELECT USING (true);

-- Clan creation goes through Edge Function (service_role)
-- Leader/officer updates go through Edge Function (service_role)
```

### `clan_memberships`

```sql
-- Anyone can read memberships (clan rosters are public)
CREATE POLICY "memberships_select" ON clan_memberships FOR SELECT USING (true);

-- Join/leave/promote goes through Edge Functions (service_role)
```

### `clan_wars`, `clan_war_contributions`

```sql
-- Anyone can read wars and contributions (public competition data)
CREATE POLICY "wars_select" ON clan_wars FOR SELECT USING (true);
CREATE POLICY "contributions_select" ON clan_war_contributions FOR SELECT USING (true);

-- All mutations are server-only (service_role)
```

### `reports`

```sql
-- Users can read their own submitted reports
CREATE POLICY "reports_select_own" ON reports FOR SELECT
  USING (auth.uid() = reporter_id);

-- Users can insert reports
CREATE POLICY "reports_insert" ON reports FOR INSERT
  WITH CHECK (auth.uid() = reporter_id);
```

### `appeals`

```sql
-- Users can read their own appeals
CREATE POLICY "appeals_select_own" ON appeals FOR SELECT
  USING (auth.uid() = user_id);

-- Users can insert appeals for their own workouts
CREATE POLICY "appeals_insert" ON appeals FOR INSERT
  WITH CHECK (
    auth.uid() = user_id
    AND workout_id IN (SELECT id FROM workouts WHERE user_id = auth.uid())
  );
```

### `cosmetic_inventory`

```sql
-- Users can read their own inventory
CREATE POLICY "cosmetics_select_own" ON cosmetic_inventory FOR SELECT
  USING (auth.uid() = user_id);

-- Others can read cosmetics for display purposes (equipped items)
CREATE POLICY "cosmetics_select_public" ON cosmetic_inventory FOR SELECT
  USING (true);

-- All mutations are server-only (service_role)
```

### `seasons`

```sql
-- Anyone can read seasons
CREATE POLICY "seasons_select" ON seasons FOR SELECT USING (true);

-- All mutations are server-only (service_role)
```

### Service Role Access

Server-derived fields are written using the Supabase `service_role` key, which bypasses RLS. This key is used exclusively in:
- Edge Functions (workout validation pipeline, scoring, war management)
- Cron jobs (season transitions, streak calculations, matchmaking)
- Webhook handlers (purchase verification)

The `service_role` key must never be exposed to the client.

## Migration Strategy

### Phase 1: Foundation (initial deployment)

1. **Create custom types (enums):**
   - `rank_tier`: bronze, silver, gold, platinum, diamond, champion
   - `subscription_tier`: free, premium
   - `workout_type`: strength, scout
   - `workout_status`: draft, submitted, validated, rejected
   - `validation_status_type`: accepted, accepted_with_low_confidence, held_for_review, excluded_from_clan_score, rejected
   - `evidence_type`: gps_route, heart_rate_log, accelerometer_summary, photo, wearable_sync, manual_note
   - `workout_source`: manual, sensor, wearable
   - `clan_role`: leader, officer, member
   - `war_status`: scheduled, active, completed
   - `report_category`: impossible_stats, suspected_spoofing, inappropriate_behavior, other
   - `report_status`: pending, investigating, resolved, dismissed
   - `appeal_status`: pending, under_review, approved, denied
   - `cosmetic_type`: badge, theme, title, clan_banner
   - `acquisition_type`: season_reward, purchase, achievement
   - `season_status`: upcoming, active, completed

2. **Create tables** in dependency order:
   - `profiles` (depends on `auth.users`)
   - `seasons`
   - `workouts` (depends on `auth.users`)
   - `workout_evidence` (depends on `workouts`)
   - `workout_validations` (depends on `workouts`)
   - `clans` (depends on `auth.users`)
   - `clan_memberships` (depends on `clans`, `auth.users`)
   - `clan_wars` (depends on `seasons`, `clans`)
   - `clan_war_contributions` (depends on `clan_wars`, `auth.users`, `clans`, `workouts`)
   - `reports` (depends on `auth.users`, `workouts`)
   - `appeals` (depends on `auth.users`, `workouts`)
   - `cosmetic_inventory` (depends on `auth.users`)

3. **Create triggers:**
   - `updated_at` auto-refresh on `profiles`, `workouts`, `clans`
   - `member_count` maintenance on `clan_memberships` insert/delete
   - Profile auto-creation on `auth.users` insert

4. **Create indexes** (as listed above).

5. **Enable RLS** on all tables and create policies.

### Phase 2: Iteration

- Use Supabase CLI (`supabase migration new <name>`) for all schema changes.
- Each migration is a single SQL file in `supabase/migrations/`.
- Never modify a migration that has been applied to production. Create a new one instead.
- Test migrations locally with `supabase db reset` before deploying.

### Phase 3: Future considerations

- **Leaderboards table:** Materialized or cached leaderboard snapshots for performance (avoid real-time aggregation on large datasets).
- **Notifications table:** In-app notifications for war results, rank changes, appeal outcomes.
- **Achievements table:** Achievement definitions and unlock conditions (currently cosmetics serve this role, but a dedicated system may be warranted).
- **Audit log:** Track admin actions on reports, appeals, and manual score adjustments.
- **Partitioning:** If `workouts` grows very large, consider range-partitioning by `created_at` (quarterly or monthly).
